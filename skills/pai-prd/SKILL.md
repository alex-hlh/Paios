# PRD — Product Requirements Document (pai:prd)

## 定位

在 `pai:bootstrap` 之后、`pai:story` 之前运行。回答"为什么要做？谁会用？成功标准？范围边界？"。

**核心产出**：
- `ai/prd.md` — 完整产品需求文档
- `ai/personas.md` — 用户画像卡片

## 前置条件

```
依赖检查:
  [ ] ai/config.yaml    → 可选。有则读取项目名称和技术栈
  [ ] ai/state/         → 可选。有则检查是否已有 prd/change 进行中
  [ ] ai/prd.md         → 存在则询问"编辑已有"还是"新建"
```

## 7 阶段工作流

使用 {task-manager} 创建以下 7 个任务，逐阶段执行。**每个阶段产出需经用户确认才能进入下一阶段。**

---

### 阶段 0: 启动与上下文感知

自动执行，不提问：

1. 检查 `ai/prd.md` 是否已存在 → 存在则询问"编辑已有 / 重新生成 / 取消"
2. 用 {search-glob} 扫描 `ai/` 下是否有 `personas.md`、`user-stories.md` 等文件
3. 用 {file-read} 读取 `ai/config.yaml`，获取项目名和技术栈
4. 输出启动横幅：

```
+============================================+
|  PRD — 产品需求文档                         |
+============================================+
|                                            |
|  项目: 二手相机交易平台                      |
|  技术栈: Node.js + React + PostgreSQL      |
|                                            |
|  已有文件: ai/config.yaml, ai/rules/       |
|  状态: 新建 PRD                            |
|                                            |
+============================================+
```

---

### 阶段 1: 问题陈述与商业目标

**AI 提问（强制，逐问等待回复）：**

1. "我们正在解决用户的什么痛点或需求？请用一句话描述。"
2. "如果不解决这个问题，用户会面临什么后果？"
3. "这个产品的商业目标是什么？（提升转化率、降低客服成本、新增收入流...）"
4. "如何定义项目成功？请给出 1-2 个可衡量的指标。"

**产出：** 写入 `ai/prd.md`「问题陈述与目标」章节：

```markdown
## 1. 问题陈述与目标

+-----------------------------------------------+
| 当前状态：用户找不到可靠的二手相机卖家。       |
| 痛点：信息不对称，交易纠纷多，时间成本高。     |
| 机会：提供验机服务 + 担保交易的平台。          |
+-----------------------------------------------+

### 成功指标
- 平台上线 6 个月 GMV 达到 50 万美元。
- 交易纠纷率 < 1%。
```

**门禁：** "以上问题陈述和成功指标无误？[Y/n]"

---

### 阶段 2: 用户画像与干系人分析

**AI 提问（逐问等待回复）：**

1. "谁是这个产品的主要用户？请至少列举 3 类角色。"
2. （对每个角色）"核心目标是什么？使用频率？技术熟练度？"
3. "还有哪些间接干系人？（客服、管理员、第三方供应商）"

**产出：** 生成 `ai/personas.md`，每个角色一张 ASCII 卡片：

```markdown
+--- Persona 1: 个人买家 Alice ---+
| 年龄：28 | 职业：设计师        |
| 目标：买到性价比高的二手相机   |
| 痛点：担心买到假货或翻新机     |
| 使用频率：每月 2-3 次          |
| 技术能力：中等                  |
+---------------------------------+

+--- Persona 2: 个人卖家 Bob -----+
| 年龄：35 | 职业：摄影师        |
| 目标：快速安全地出售闲置设备    |
| 痛点：怕遇到到手刀、被压价     |
| 使用频率：每季度 1-2 次         |
| 技术能力：高                    |
+---------------------------------+

+--- Persona 3: 平台运营 Carol ---+
| 年龄：30 | 职业：运营经理      |
| 目标：提升平台交易量和信任度   |
| 痛点：交易纠纷处理成本高       |
| 使用频率：每日                 |
| 技术能力：低（使用后台工具）    |
+---------------------------------+
```

同时在 `ai/prd.md` 中引用：

```markdown
## 2. 用户画像

详见 [personas.md](ai/personas.md)。

共识别 3 类角色：个人买家、个人卖家、平台运营。
```

**门禁：** "角色清单是否完整？有需要补充的吗？[Y/n]"

### 阶段 3: 用户旅程（To-Be）

**AI 操作：**

1. 选取一个核心角色 + 关键场景（如"购买商品"）
2. 用 ASCII 泳道图绘制**未来用户旅程（To-Be）**

**产出：** 写入 `ai/prd.md`「用户旅程」章节：

```markdown
## 3. 用户旅程（To-Be）

场景：购买商品（角色：个人买家 Alice）

用户 Alice            平台                 卖家 Bob
│                      │                   │
│ 搜索"索尼A7III"     │                   │
│─────────────────────→│                   │
│                      │ 返回搜索结果      │
│←─────────────────────│                   │
│                      │                   │
│ 点击商品             │                   │
│─────────────────────→│                   │
│                      │ 查询详情+验机报告 │
│                      │──────────────────→│
│                      │←── 报告 + 价格 ──│
│ 查看详情 + 报告      │                   │
│←─────────────────────│                   │
│                      │                   │
│ 发起下单             │                   │
│─────────────────────→│                   │
│                      │ 锁定商品，创建订单│
│                      │──────────────────→│
```

**门禁：** "这个旅程是否准确反映了产品要提供的核心体验？[Y/n]"

### 阶段 4: 功能范围与 MoSCoW 优先级

**AI 提问（逐问等待回复）：**

1. "为了实现上面的用户旅程，需要哪些核心功能？请头脑风暴列举。"
2. "对每个功能，按 MoSCoW 打标：Must have / Should have / Could have / Won't have"

**产出：** 写入 `ai/prd.md`「功能范围与优先级」章节：

```markdown
## 4. 功能范围与优先级

| 功能 | Must | Should | Could | Won't |
|------|:----:|:------:|:-----:|:-----:|
| 商品列表与搜索 | X |  |  |  |
| 商品详情与验机报告 | X |  |  |  |
| 用户注册/登录 | X |  |  |  |
| 担保交易 | X |  |  |  |
| IM 即时通讯 |  | X |  |  |
| 卖家认证 |  | X |  |  |
| 社区功能 |  |  | X |  |
| 直播带货 |  |  |  | X （原因：v1 资源不足）|

### MVP 范围

Must have 功能构成 MVP，是第一个发布版本的全部内容。
```

**门禁：** "功能分类确认，尤其是 MVP（Must have）边界是否准确？[Y/n]"

### 阶段 5: 非功能需求（NFR）

**AI 操作：**

从安全、性能、可用性、可维护性、兼容性五个维度逐项引导。每条 NFR 必须是可验证的（含具体数字或测试方法）。

检查 AIOS 已有规则，标注"已被 X 规则覆盖"避免重复。

**产出：** 写入 `ai/prd.md`「非功能需求」章节：

```markdown
## 5. 非功能需求

| 类别 | 需求描述 | 验证方式 |
|------|---------|---------|
| 性能 | 商品列表 API p95 响应 < 200ms | 压力测试 |
| 性能 | 页面首屏加载 < 1.5s (3G 网络) | Lighthouse |
| 安全 | 支付流程符合 PCI-DSS 要求 | 安全审计 |
| 安全 | 密码哈希使用 bcrypt（已被 ai/rules/ 的 SEC004 覆盖） | 代码审查 |
| 可用性 | 主要流程在 3 次点击内可达 | 用户测试 |
| 兼容性 | 支持 Chrome/Safari/Firefox 最近 2 个版本 | 自动化 E2E |
```

**门禁：** "NFR 列表确认无误？这些指标将作为架构设计的约束。[Y/n]"

---

### 阶段 6: 里程碑与发布计划

**AI 操作：**

基于 MoSCoW 结果提出 2-3 个版本里程碑。每个里程碑包含目标、功能列表、预计时间窗口。

**产出：** 写入 `ai/prd.md`「里程碑」章节：

```markdown
## 6. 里程碑与发布计划

v0.1 MVP         v0.2               v1.0
├────────────────┼──────────────────┼──────────→
目标：验证      目标：提升信任      目标：生态
核心交易流程    卖家认证系统        社区 + 积分
商品浏览        即时通讯            直播试水
基础用户系统    评价体系

### v0.1 (MVP) — 2-4 周
商品列表与搜索、商品详情、用户注册登录、担保交易

### v0.2 — 4-6 周
IM 通讯、卖家认证、评价体系

### v1.0 — 8-12 周
社区功能、积分系统、直播带货（视资源）
```

**门禁：** "版本节奏是否符合业务需求和资源现状？[Y/n]"

---

### 阶段 7: 约束、假设与风险

**AI 提问（逐问等待回复）：**

1. "项目有什么已知约束？如时间、团队规模、第三方依赖。"
2. "我们做了什么关键假设？如'卖家会主动上传验机报告'。"
3. "最大的风险是什么？如果假设不成立怎么办？"

**产出：** 写入 `ai/prd.md`「约束、假设与风险」章节：

```markdown
## 7. 约束、假设与风险

+---------------------------------------------+
| 约束:                                       |
| - 团队 1 后端 + 1 前端 + 1 兼职设计师       |
| - 必须使用 AWS 现有账号，不能新增云服务商    |
| - 上线期限：12 周                           |
|                                             |
| 假设:                                       |
| - 卖家愿意上传验机报告（如不成立→平台代验）|
|                                             |
| 风险:                                       |
| ⚠️ 担保交易涉及资金托管，合规成本可能超预期 |
| ⚠️ 二手商品标准化难度高，验机报告质量不稳定  |
+---------------------------------------------+
```

**最后门禁：**

"PRD 全部内容已完成，请通读确认。"

## 与 pai:story 的衔接

`pai:prd` 完成后，在 `ai/prd.md` 末尾自动生成"传递给 pai:story"区块：

```
---story-input
## 传递给 pai:story 的输入

- MVP 范围：Must have 功能列表（4 项）
- 核心用户角色：从 ai/personas.md 提取（3 类）
- 关键旅程：To-Be 购买商品旅程
---story-input
```

这样 `pai:story` 可以直接识别这些输入，从 MVP 功能开始拆解用户故事和原型。

## 注意事项

- **不替代 pai:story** — PRD 回答"做什么、为什么做"，Story 回答"怎么做、交互细节"
- **不超过 7 阶段** — 每个阶段用户必须确认后才能进入下一阶段
- **范围蔓延防控** — MoSCoW 的 Won't have 要明确记录原因
- **可回溯** — 任何时候都可以重新运行 pai:prd 编辑已有 ai/prd.md
- **NFR 不重复规则** — AIOS 已有规则标注引用（如 SEC003 覆盖 SQL 注入），避免冗余
