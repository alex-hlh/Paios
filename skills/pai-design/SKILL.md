# Design (pai:design)

## Standalone Usage

`pai:design` 可以独立使用，即使项目未初始化也能工作。

```
依赖检查 (独立调用):
  [ ] ai/config.yaml    → 可选。有则遵循项目规范，无则使用 universal 预设
  [ ] ai/rules/         → 可选。有则在设计中已知约束，无则仅提醒通用原则
  [ ] ai/specs/         → 可选。有则检查是否需要 MODIFIED，无则假设全部 ADDED
  [ ] ai/changes/       → 可选。无则自动创建 ai/changes/ 目录

降级行为:
  - 无 ai/ 目录 → 仍可完成设计讨论，写入时询问用户输出路径
  - 无 ai/config.yaml → 默认: conventional commits, 2空格, double引号, kebab-case
  - 无 ai/specs/ → 提示 "这是首个 spec，所有需求均为 ADDED"
```

## 核心规则

<HARD-GATE>
在用户明确批准设计之前，不得调用任何实现技能（pai:spec, pai:build）、编写任何代码或采取任何实现行动。这适用于所有项目，无论多简单。
</HARD-GATE>

## 流程

使用 {task-manager} 创建以下任务并逐个完成：

### 阶段 0: 检测是否有已有计划书

**如果用户提供了项目计划书文档**（如 PRD、技术方案、需求文档链接或文件路径），**跳过阶段 1-2 的发现式提问**，改为导入模式：

**0A. 读取计划书**
- 使用 {file-read} 或 {web-fetch} 读取用户提供的文档
- 分析文档结构，提取以下信息：

| 信息 | 文档中的对应段落 |
|------|-----------------|
| 项目目标/背景 | "背景" / "目标" / "概述" / "Purpose" |
| 功能范围 | "功能需求" / "需求列表" / "Scope" / "Features" |
| 技术约束 | "技术栈" / "限制" / "Platform" / "Constraints" |
| 成功标准 | "交付标准" / "验收" / "Success Metrics" |
| 非功能需求 | "性能" / "安全" / "并发" / "NFR" |
| 用户角色 | "用户" / "角色" / "Persona" / "Stakeholders" |

**0B. 生成计划书摘要**

```
+============================================+
|  PLAN IMPORTED                              |
+============================================+
|  Source: PRD-2026-user-platform.pdf        |
|                                            |
|  Goal:     构建用户管理与协作平台             |
|  Users:    3 类（管理员/编辑/访客）           |
|  Features: 登录 / 权限 / 协作编辑 / 导出     |
|  Stack:    Node.js + React + PostgreSQL     |
|  Deadline: Q3 2026                          |
|                                            |
|  Extracted: 4 domains, ~12 features         |
+============================================+
```

**0C. 确认式提问（替代发现式提问）**

计划书已有大部分答案，只问缺失的部分（最多 3 问）：

```
从计划书中已提取大部分需求，只需确认以下:

Q1: 计划书中提到"协作编辑"，具体方式是？
    [A] 实时协同 (类似 Google Docs)
    [B] 评论+修订 (类似 Word 审阅)
    [C] 锁定编辑 (一次一人)

Q2: 导出格式支持哪些？[PDF / Word / Markdown / 全选]
```

**0D. 进入阶段 3（跳过阶段 1-2 的发现式提问）**

通过确认式提问补充缺失信息后，直接进入方案探索阶段。

### 阶段 1: 探索项目上下文（无计划书时执行）
- 使用 {search-glob} 和 {file-read} 检查项目现有文件、文档
- 读取 `ai/` 目录下的当前状态和已有 specs
- 读取 `ai/config.yaml` 了解技术栈和规范
- 读取 `ai/rules/` 了解架构和风格约束

### 阶段 2: 需求澄清（一问一答。有计划书时任可跳过，阶段 0 已提取）

**提问规则：**

1. 一次只问一个问题，等待用户回复后再继续
2. 优先使用选择题（2-4 个选项），开放问题作为备选
3. 每个问题只聚焦一个话题，如需深入则拆为多轮
4. 每个问题必须是你**无法从已有上下文中推断**的信息——不要问你已经知道答案的问题

**必问的 5 个维度（按顺序覆盖，各维度至少 1 问）：**

| # | 维度 | 目的 | 典型问法 |
|---|------|------|---------|
| 1 | **目的** | 理解为什么做、解决什么用户问题 | "这个功能主要解决什么用户痛点？" |
| 2 | **范围** | 明确做什么、不做什么 | "以下哪些功能在范围内？A) XX B) XX C) XX" |
| 3 | **约束** | 技术栈限制、性能要求、兼容性 | "对响应时间有要求吗？需要兼容旧版吗？" |
| 4 | **成功标准** | 如何判断做完了、做对了 | "完成的标准是？A) 能跑通 B) 有测试 C) 已部署" |
| 5 | **边界/异常** | 极端情况、错误处理期望 | "如果用户输入错误密码 3 次，期望什么行为？" |

**提问数量指南：**

| 变更规模 | 建议提问数 | 说明 |
|---------|-----------|------|
| 小（改配置/加按钮/换颜色） | 2-3 问 | 跳过维度 5，直接进入阶段 4（设计可以很短） |
| 中（加一个功能模块） | 4-6 问 | 覆盖全部 5 个维度，维度 2/4 各 1-2 问 |
| 大（子系统/新服务） | 6-10 问 | 维度 1-5 各 1-2 问，先拆解再逐个子项目设计 |

**停止条件（同时满足以下 3 条才能进入阶段 3）：**

- [ ] 5 个必问维度中，至少 4 个得到了明确回答
- [ ] 你能用一句话清晰描述：做什么、为谁做、成功标准是什么
- [ ] 不存在"这取决于用户没说清楚的假设"——所有关键假设已被确认或排除

**反模式——以下情况说明还没问够：**

| 如果你这么想 | 说明还没问够 |
|-------------|------------|
| "我大概知道用户要什么了" | 模糊的理解不够，用选择题确认具体细节 |
| "这个很简单，不需要再问了" | 简单的事最容易出误解，至少问完目的+范围 |
| "用户好像不确定，我先做个原型" | 不确定恰恰说明需要更多澄清，用选项帮用户想清楚 |
| "我可以自己假设这个细节" | 任何假设都必须向用户确认，不要替用户决定 |

如果需求涉及多个独立子系统（如 "做一个包含聊天、文件存储、支付的平台"），**不要在阶段 2 逐个细化各子系统**。直接标记并帮助用户拆解为子项目，然后对第一个子项目执行完整的 design → spec → build 流程。

### 阶段 3: 方案探索
- 提出 2-3 种不同方案，含利弊对比
- 先展示推荐方案及其理由
- 让用户选择方向

### 阶段 4: 分节展示设计

展示设计时**必须使用 ASCII 图表辅助可视化**，不要只靠文字描述。以下是各类图表的模板：

**架构图模板：**
```
+------------------+     +------------------+     +------------------+
|   Browser/Client  |     |    API Gateway    |     |   Auth Service    |
|  (React/Next.js)  |---->|  (Express/FastAPI)|---->|   (JWT/Session)   |
+------------------+     +------------------+     +------------------+
         |                        |                          |
         |                        v                          v
         |               +------------------+     +------------------+
         |               |   App Server     |     |   DB / Cache     |
         +-------------->|  (Business Logic) |---->|  (PostgreSQL +   |
                         +------------------+     |   Redis)         |
                                                  +------------------+
```

**数据流图模板：**
```
User Action            API Request            Service              Response
    |                     |                      |                    |
    |  1. POST /login     |                      |                    |
    |-------------------->|                      |                    |
    |                     |  2. validate + query  |                    |

    |                     |                      |  3. query DB       |
    |                     |                      |------------------>|

    |                     |                      |<------------------|

    |                     |<---------------------|                    |
    |  6. 200 { user,     |                      |                    |

```

**组件/模块关系图模板：**
```

|   +-- Header/          |   +-- Login/          |   +-- useAuth/
```

**任务依赖图（复杂场景）:**
```
   [User Model] ──→ [Auth Logic] ──→ [Login API] ──→ [Integration Test]
        │                                                   │
        └──→ [Token Utility] ──────→ [Middleware] ──────────┘
```

**使用规则：**
- 每个设计节**至少包含一个 ASCII 图**（架构图、数据流图或模块关系图）
- 图放在文字之前："先看图，再看文字"
- 设计确认时，逐节展示："这部分看起来对吗？"
- 每节 200-300 字 + 1 个 ASCII 图
- 覆盖：架构、组件、数据流、错误处理、测试

### 阶段 5: 写入设计文档
用户确认后，写入 `ai/changes/<change-name>/`：

**proposal.md**:
```markdown
# Proposal: <title>

## Intent
<为什么做这个>

## Scope
在范围内:
- <item>
不在范围内:
- <item>

## Approach
<技术方案概述>
```

**design.md**:
```markdown
# Design: <title>

## Technical Approach
<技术方案详情>

## Architecture Decisions
<关键决策及原因>

## Data Flow
<数据流描述>

## File Changes
- <受影响的文件列表>
```

### 阶段 6: 设计文档自检
写入后自查：
1. 是否有占位符(TBD/TODO)或未完成部分？
2. 是否有内部矛盾？
3. 范围是否在一个实现计划内？
4. 是否有模糊需求（两种不同解读）？

### 阶段 7: 用户审查
提示用户审查 `ai/changes/<change-name>/proposal.md` 和 `design.md`，确认后进入下一阶段。

## 需求变更处理（开发中途新增功能）

> **推荐入口**: 开发中途的需求变更，优先使用 `/pai:amend` — 它会检测当前 change 状态并推荐策略。本节仅在 `pai:amend` 不可用时作为后备。

任何时候用户提出新需求，先检查当前是否已有活跃 change：

### 策略 1: 追加到当前 Change（小变更，同领域）

新需求与当前 change 高度相关 → 更新已有文档 + 重新生成 spec：

```
1. 更新 proposal.md → 追加 scope 条目
2. 更新 design.md → 补充新增模块设计
3. 追加 `[ ] <新 task>` 到现有 tasks.md（或重新运行 pai:spec）
4. 继续 pai:build

适用: "登录功能再加个验证码" / "用户模型加个头像字段"
```

### 策略 2: 新建并行 Change（独立功能）

新需求独立于当前 work → 创建新 change，并行推进：

```
1. 用 {file-edit} 在 ai/state/current.md 标记当前 change 进度
2. 启动新的 pai:design → 生成独立 change 目录
3. 两个 change 并行：ai/state/current.md 记录多 change
4. 归档时 pai:done 自动检测 spec 冲突

适用: 正在做登录，又接到"做权限管理" / "做缓存层"
```

### 策略 3: 延期（超大 / 不确定）

新需求明确但规模过大 → 记录到 roadmap，不立即启动：

```
1. 用 {file-edit} 追加到 ai/state/roadmap.md 下一版本
2. 继续当前 change

适用: "顺便把整个权限系统重构" / "这个功能需要微服务拆分"
```

### 用户不确定时：询问

```
+============================================+
|  SCOPE CHANGE                              |
+============================================+
|  Current:  add-user-login [3/8 tasks done] |
|  New:      add-2fa                         |
|                                            |
|  How to handle?                            |
|  A) Same domain → append to current change |
|  B) Independent → new parallel change       |
|  C) Defer → record in roadmap              |
+============================================+
```

## 设计原则

- **隔离与清晰**: 将系统拆为小单元，每个单元有明确目的、定义良好的接口、可独立测试
- **YAGNI**: 删除不必要的功能
- **在现有代码库中工作**: 探索现有结构后再提方案，遵循既有模式，不做无关重构
- **遵循配置**: 遵循 `ai/config.yaml` 中的 conventions（命名、代码风格等）

## 完成后

设计文档确认后，触发 `pai:spec` 生成 delta spec 和 tasks。
