# Story — 需求分析与原型设计 (pai:story)

## 定位

在 `pai:prd` 之后、`pai:design` 之前运行。将 PRD 中的功能需求转化为**可交互的原型图 → 数据流 → 后台流程 → 接口列表 → 前后端功能清单**。

## 前置条件

```
依赖检查:
  [ ] prd.md      → 推荐。有则读取功能范围和用户画像
  [ ] personas.md → 推荐。提取关键角色
  [ ] ai/config.yaml → 可选。获取技术栈信息

降级行为:
  - 无 prd.md → 询问用户直接描述要分析的功能
  - 无 personas.md → 跳过角色映射，仅分析功能本身
```

## 产出物

| 文件 | 内容 |
|:---|:---|
| `ai/prototype.md` | ASCII 原型图（页面线框图） |
| `ai/data-flows.md` | 页面元素交互与数据流 |
| `ai/backend-flows.md` | 后台业务流程图 |
| `ai/api-list.md` | 接口清单与功能描述 |
| `ai/feature-points.md` | 前后端全部功能点 |

## 5 步工作流

使用 {task-manager} 创建 5 个任务，逐步执行。**每步产出需经用户确认才能进入下一步**。

> **新功能发现处理**：在任何步骤中，如果用户提出超出当前范围的新功能，遵循 Step 1 的路径 B 规则——不打断当前流程，记录到 `ai/feature-points.md` 的待定区，在当前 story 完成后单独处理。

### 步骤 0: 上下文感知

2. 用 {search-glob} 检查 `docs/` 下是否已有 prototype/data-flows 等文件
3. 询问用户："本次分析哪个功能？"（可从 PRD 的 Must have 列表中选择）
4. 输出启动横幅：

|                                            |

|    5. 功能点汇总     → ai/feature-points.md  |
|                                            |

```

### 第一步：生成 ASCII 原型图

**目标**：用 ASCII 绘制核心页面的线框图。

**规则**：
- 必须逐页绘制，一次只画一个页面
- 画完一页必须展示给用户确认
- 用户不同意则修改后再确认
- 确认后写入 `ai/prototype.md`

**流程**：

1. 问第一个问题：
```
本功能涉及几个页面？
列出每个页面的名称和用途。
例：登录页（用户输入凭据）、注册页（创建新账号）
```

用户回答后：

2. 每页画图前追问细节（一次一问）：
```
关于"登录页":
- 页面顶部有什么？（导航栏/Logo/链接）
- 中间主体有哪些输入框？分别是输入什么？
- 有哪些按钮/操作链接？
- 页面底部有何内容？
```

3. 基于答案生成 ASCII 原型并展示：

```
+--------------------------------------------------+
|  [Logo]                          [帮助] [注册]    |
+--------------------------------------------------+
|                                                    |
|                 欢迎回来                           |
|                                                    |
|          +----------------------------+            |
|          |  邮箱 / 手机号              |            |
|          +----------------------------+            |
|          +----------------------------+            |
|          |  密码                       |            |
|          +----------------------------+            |
|                                                    |
|          [ 忘记密码？ ]                            |
|          [     登录按钮      ]                      |
|                                                    |
+--------------------------------------------------+

这个登录页布局正确吗？有哪些需要调整？
```

4. **用户反馈处理——两种路径**：

当用户在看原型时提出新增内容，先判断是"细节补充"还是"新功能发现"：

```
路径 A — 细节补充（在原图范围内）
─────────────────────────────
你: "加个'记住我'复选框"
→ 判定: 同页面内的 UI 元素扩展
→ 直接修改原型图，不改变 scope
→ 修改后重新展示，继续确认循环

路径 B — 新功能发现（超出原图范围）
─────────────────────────────
你: "再加个生物识别登录页面"
→ 判定: 需要新建页面/新增 API/新增 domain
→ 不直接修改当前原型，改为提问:
  "这个属于当前 MVP 范围吗？
   A) 是 → 追加到当前功能分析
       增加新页面，后面步骤会为它补充数据流和接口
   B) 否 → 记录为'新发现'，但不阻塞当前流程
       记入 feature-points.md 的"待定"区，
       在当前 story 完成后单独处理"
```

**判定标准**：

| 用户说的 | 路径 | 原因 |
|---------|:----:|------|
| "加个输入框" | A | 同一页面内的 UI 元素 |
| "加个复选框" | A | 同一页面内的 UI 元素 |
| "加个跳转链接" | A | 页面内的交互元素 |
| "加个新页面" | B | 需要新建页面 |
| "再加个第三方支付" | B | 新功能域，新 API |
| "这里再加个筛选功能" | B | 独立的功能模块 |

路径 A 直接在循环内处理，路径 B 记录到 `ai/feature-points.md` 的待定区：

```markdown
## 待定功能（当前轮次不分析）

以下功能在原型确认过程中发现，已记录但未在当前 story 中处理：

| 功能 | 发现阶段 | 建议 |
|------|---------|------|
| 生物识别登录 | 登录页原型 | 可在后继 sprint 中规划 |
```

```
你: 加一个"记住我"复选框

AI: 已更新登录页原型，添加了"记住我"复选框。

+--------------------------------------------------+
...
|          |  [记住我]                  |            |
...
现在正确吗？[Y/n]
```

5. **门禁**：所有页面已确认。如有路径 B 发现的功能，提示用户待处理数量。

```
已完成页面原型:
  ✓ 登录页 (已确认)
  ✓ 注册页 (已确认)
  ⏳ 待定: 1 个功能（生物识别登录）记录在 feature-points.md
进入下一步。
```

### 第二步：绘制可操作点的数据流

**目标**：为原型图上的每个可操作元素标注"触发什么动作、携带什么数据、期望什么响应"。

**规则**：
- 一次只处理一个可操作元素
- 每个元素必须经过"用户描述 → AI 生成 → 用户确认"循环
- 所有元素确认后一次性写入 `ai/data-flows.md`

**流程**：

1. 从已确认的 `ai/prototype.md` 中提取可操作元素清单并展示：

```
从原型图中识别到以下可操作元素:
登录页:
  1. 邮箱输入框
  2. 密码输入框
  3. 记住我复选框
  4. 登录按钮
  5. 忘记密码链接
  6. 注册跳转链接
  7. Google 登录按钮
  8. GitHub 登录按钮

从第一个元素开始：
```

2. 对每个元素逐轮追问 → 生成 → 确认：

```
元素 1: 邮箱输入框
当用户输入邮箱后点击登录按钮，前端如何处理这个输入？
- 输入什么数据？  [email: string]
- 需要什么格式校验？  [邮箱格式/非空]
- 校验不通过时如何展示？ [红色边框 + "请输入有效邮箱"]

元素 2: 登录按钮
点击登录按钮触发什么？
- 调用哪个接口？  [POST /api/auth/login]
- 传入什么参数？  [{ email, password }]
- 成功后返回什么？  [{ token, user }]
- 前端如何处理成功响应？ [存储 token，跳转首页]
- 失败后如何处理？  [显示错误提示，输入框保留]
```

3. 每个元素确认后记录，然后进入下一个

4. 所有元素确认后，生成完整数据流并输出到 `ai/data-flows.md`：

```
## 登录页数据流

[邮箱输入框] → email: string → 校验: 非空+邮箱格式
[密码输入框] → password: string → 校验: 非空
[记住我]     → remember: boolean → 传递给 login 接口
[登录按钮] ──→ POST /api/auth/login({email, password, remember})
                ├─ 200 → 存储 token → 跳转首页
                └─ 401 → 显示"邮箱或密码错误"

## 注册页数据流
...
```

5. **门禁**：所有元素已确认，数据流完整。[Y/n]

### 第三步：给出后台流程

**目标**：从后端视角，用 ASCII 泳道图描述每个用户操作的业务处理步骤。

**规则**：
- 一次只处理一个 API/操作
- 每个流程必须覆盖成功路径和所有异常路径
- 每完成一个流程即展示给用户确认

**流程**：

1. 从 `ai/api-list.md`（如已存在）或第二步结果中提取 API 清单：

```
需要绘制后台流程的 API:
  1. POST /api/auth/login
  2. POST /api/auth/register

从第一个开始：
```

2. 逐个 API 执行问-答-绘循环：

```
API 1: POST /api/auth/login

> 后端收到请求后第一步做什么？
  你: [验证邮箱格式和密码是否为空]

> 验证通过后做什么？
  你: [查询数据库，查找该邮箱对应的用户]

> 查到用户后做什么？
  你: [用 bcrypt 比对密码]

> 密码正确后做什么？
  你: [生成 JWT token，返回客户端]

> 各种异常分支如何处理？
  你: 邮箱不存在 → 返回 401 "用户不存在"
      密码错误 → 返回 401 "密码错误"
      连续5次错误 → 锁定账号10分钟 → 返回 423
      服务器错误 → 返回 500

> 总结后展示泳道图：
```

3. 展示泳道图并确认：

```
客户端                   后端 API             数据库
  │                       │                    │
  │ POST /api/auth/login  │                    │
  │ {email, password}     │                    │
  │──────────────────────→│                    │
  │                       │ 参数校验            │
  │                       ├─ 必填+格式          │
  │                       │                    │
  │                       │ 查用户 by email    │
  │                       │───────────────────→│
  │                       │←── 存在? ───────────│
  │                       │    │               │
  │                       │    ├─ 不存在        │
  │                       │    │ → 401 err     │
  │  ←─ 401 ──────────────│←───┘               │
  │                       │    │               │
  │                       │    └─ 存在          │
  │                       │    ├─ bcrypt 比对密码│
  │                       │    │  ├─ 错误        │
  │                       │    │  │ → 401      │
  │  ←─ 401 ──────────────│←────┘              │
  │                       │    │  └─ 正确        │
  │                       │    │  └─ 生成 JWT   │
  │  ←─ 200 {token} ─────│←───────────────────│

正确吗？[Y/n]
```

4. 用户确认后追加到 `ai/backend-flows.md`，进入下一个 API

5. 所有 API 完成后输出摘要

**门禁**：所有后台流程已确认。[Y/n]

### 第四步：定义接口与后端功能

**目标**：将后台流程中的关键节点转化为具体的 REST API，形成接口文档初稿。

**规则**：
- 一次只定义一个 API 端点
- 必须覆盖请求/响应结构、状态码、关联功能
- 逐个确认后写入 `ai/api-list.md`

**流程**：

1. 从已确认的 `ai/backend-flows.md` 中提取 API 清单：

```
从后台流程中识别出以下接口:
  1. POST /api/auth/login
  2. POST /api/auth/register

从第一个开始：
```

2. 逐个 API 问-答-写循环：

```
POST /api/auth/login

> 这个接口有哪些请求参数？哪个是必填？
  你: email: string (必填), password: string (必填)

> 成功时返回什么格式？
  你: { token: string, user: { id, nickname, email } }

> 可能的失败响应有哪些？
  你: 401 { error: "邮箱或密码错误" }
      423 { error: "账号已锁定", retryAfter: 600 }
      500 { error: "服务器内部错误" }

> 关联的后端模块和方法？
  你: AuthService.login(), UserRepository.findByEmail()
```

3. 展示完整接口定义并确认：

```
### POST /api/auth/login
- 描述: 用户登录
- 请求: { email: string, password: string }
- 成功 200: { token, user: { id, nickname, email } }
- 失败 401: { error }
- 失败 423: { error, retryAfter }
- 关联: AuthService.login(), UserRepository.findByEmail()

正确吗？[Y/n]
```

4. 用户确认后追加到 `ai/api-list.md`，进入下一个接口

5. 所有接口确认后，汇总后端功能模块：

```
后端功能模块汇总（从 API + 流程提取）:
  AuthService: login(), register(), refreshToken()
  UserRepository: findByEmail(), createUser()
  TokenUtil: generateJWT(), verifyJWT()
  RateLimiter: checkLoginAttempts(), resetAttempts()

这些模块正确吗？[Y/n]
```

**门禁**：所有接口和后端模块已确认。[Y/n]

### 第五步：汇总前后端完整功能点

**目标**：生成可供 `pai:spec` 和 `pai:design` 直接使用的功能清单。

**规则**：
- 自动从前面 4 步的产出中提取，无需额外提问
- 但必须逐类展示并让用户确认

**流程**：

1. 展示前端功能点：

```
从原型图和数据流中提取的前端功能点:

├─ 页面
│  ├─ 登录页 (邮箱/密码/记住我/提交/忘记密码/三方登录)

│  ├─ 表单前端校验 (空值/邮箱格式/密码一致性)
│  ├─ API 调用 (login / register)
│  └─ 状态处理 (成功跳转 / 失败提示)
└─ 存储

```

2. 用户确认后，展示后端功能点：

```
从接口和后台流程中提取的后端功能点:

├─ API
│  ├─ POST /api/auth/login      (公开)
│  └─ POST /api/auth/register   (公开)
├─ 业务
│  ├─ bcrypt 密码验证
│  ├─ JWT 生成 (HS256, 24h)
│  └─ 登录失败次数限制
├─ 数据
│  └─ User { id, nickname, email, password_hash,
│            login_attempts, locked_until, created_at }
└─ 安全
   ├─ 密码强度 (8+字符, 大小写+数字)
   └─ 频率限制 (每分钟 10 次)

后端功能点正确完整吗？[Y/n]
```

3. 确认后生成功能追溯表并写入 `ai/feature-points.md`：

```
| 功能点 | 来源 PRD | MoSCoW | 关联 Story |
|--------|---------|:-----:|:----------:|
| 登录页 UI | 用户认证 | Must | US-01 |
| 注册页 UI | 用户认证 | Must | US-02 |
| JWT 认证 | 用户认证 | Must | US-03 |

```

**门禁**：所有功能点已确认，功能追溯表完整。[Y/n]

## 与 pai:design / pai:spec 的衔接

`pai:story` 完成后，`pai:design` 的阶段 0 自动读取：

- `ai/feature-points.md` → 导入功能点列表，跳过发现式提问
- `ai/api-list.md` → 作为 `pai:spec` 的 API 层 delta spec 输入

后续技能链中的 `pai:spec` 可直接从功能点生成 ADDED requirements：

```markdown
## ADDED Requirements

### Requirement: 用户登录
系统 MUST 提供邮箱+密码登录。
（来源: pai:story feature-points.md → 前端登录页 + 后端 POST /api/auth/login）
```

## 局部迭代

如需修改某一步（如仅更新原型图），通过 `/pai:amend` 管理变更范围。如果功能点有变，可指定步骤编号重新运行（如"重新做第三步"）。

## 注意事项

- **不替代 pai:design** — Story 产出业务需求和原型，Design 产出技术方案
- **不录入代码** — 所有产出都是文档，不生成任何代码
- **可选择性执行** — 用户可指定只做"原型+接口"或只做"功能点"
- **渐进式细化** — 第一次可粗粒度，后续迭代逐步细化
