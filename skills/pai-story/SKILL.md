# Story — 需求分析与原型设计 (pai:story)

## 定位

在 `pai:prd` 之后、`pai:design` 之前运行。将 PRD 中的功能需求转化为**可交互的原型图 → 数据流 → 后台流程 → 接口列表 → 前后端功能清单**。

## 前置条件

```
依赖检查:
  [ ] prd.md      → 推荐。有则读取功能范围和用户画像
  [ ] personas.md → 推荐。提取关键角色
  [ ] ai/config.yaml → 可选。获取技术栈信息

降级行为:
  - 无 prd.md → 询问用户直接描述要分析的功能
  - 无 personas.md → 跳过角色映射，仅分析功能本身
```

## 产出物

| 文件 | 内容 |
|:---|:---|
| `docs/prototype.md` | ASCII 原型图（页面线框图） |
| `docs/data-flows.md` | 页面元素交互与数据流 |
| `docs/backend-flows.md` | 后台业务流程图 |
| `docs/api-list.md` | 接口清单与功能描述 |
| `docs/feature-points.md` | 前后端全部功能点 |

## 5 步工作流

使用 {task-manager} 创建 5 个任务，逐步执行。**每步产出需经用户确认才能进入下一步**。

---

### 步骤 0: 上下文感知

自动执行，不提问：

1. 用 {file-read} 尝试读取 `prd.md` → 提取功能范围和用户角色
2. 用 {search-glob} 检查 `docs/` 下是否已有 prototype/data-flows 等文件
3. 询问用户："本次分析哪个功能？"（可从 PRD 的 Must have 列表中选择）
4. 输出启动横幅：

```
+============================================+
|  pai:story — 需求分析与原型设计             |
+============================================+
|                                            |
|  功能来源: prd.md (已存在)                  |
|  MVP 范围: 4 个 Must have                  |
|  用户角色: 买家 / 卖家 / 运营              |
|                                            |
|  本次分析: 用户登录与注册                  |
|                                            |
|  将产出:                                    |
|    1. ASCII 原型图  → docs/prototype.md   |
|    2. 数据流描述     → docs/data-flows.md  |
|    3. 后台流程       → docs/backend-flows.md|
|    4. 接口清单       → docs/api-list.md    |
|    5. 功能点汇总     → docs/feature-points.md  |
|                                            |
+============================================+
```

### 第一步：生成 ASCII 原型图

**目标**：用 ASCII 绘制核心页面的线框图，让所有干系人对"用户会看到什么、能点什么"达成一致。

**流程**：
1. 从 `prd.md` 或用户描述中提取当前功能场景
2. 询问："这个功能涉及几个关键页面？页面间跳转关系？"
3. 对每个页面生成 ASCII 原型，包含：
   - 页面布局（顶部导航、主体、底部）
   - 可操作组件（按钮、输入框、链接、下拉菜单）
   - 状态展示（加载中、空数据、错误提示）

**产出** — `docs/prototype.md`：

### 第二步：绘制可操作点的数据流

**目标**：为原型图上的每个可操作元素标注"触发什么动作、携带什么数据、期望什么响应"。

**流程**：
1. 扫描 `docs/prototype.md`，列出所有可操作元素
2. 对每个元素描述：触发动作、输入数据、预期反馈
3. 用 ASCII 流程图标明数据流向

**产出** — `docs/data-flows.md`：

```markdown
# 数据流

## 登录页数据流

用户操作：在邮箱输入框输入 "user@example.com"
         在密码输入框输入 "****"
         点击 [登录按钮]

[邮箱输入框] → 收集值 email
[密码输入框] → 收集值 password

[登录按钮] ──点击──→ 调用 login(email, password)
                         │
                         ├─ 成功 → 返回 { token, user }
                         │        → 前端存储 token 到 localStorage
                         │        → 跳转至首页
                         │        → 更新全局用户状态
                         │
                         └─ 失败 → 返回 { error: "Invalid credentials" }
                                  → 前端显示错误提示
                                  → 密码输入框保留

## 注册页数据流

用户操作：在昵称输入框输入 "Alice"
         在邮箱输入框输入 "alice@example.com"
         在密码输入框输入 "secret123"
         在确认密码输入框输入 "secret123"
         点击 [注册按钮]

[注册按钮] ──点击──→ 前端校验 (必填, 邮箱格式, 密码一致)
                         │
                         ├─ 校验未通过 → 显示具体字段错误
                         │
                         └─ 校验通过 → 调用 register({nickname, email, password})
                                           │
                                           ├─ 成功 → 返回 { token, user }
                                           │        → 自动登录
                                           │        → 跳转至首页
                                           │
                                           └─ 失败 → 返回 { error }
                                                    → 显示错误提示
```

**强制要求**：每个可操作点都必须有对应的数据流描述，不允许遗漏。

**门禁**：用户确认数据流覆盖所有交互后进入下一步。

### 第三步：给出后台流程

**目标**：从后端视角，用 ASCII 泳道图描述每个用户操作的业务处理步骤。

**流程**：
1. 基于第二步的数据流，将每个前端动作映射为后端流程
2. 用 ASCII 泳道图描述：请求入口 → 业务逻辑 → 数据库 → 外部服务 → 响应
3. 明确成功路径和异常路径

**产出** — `docs/backend-flows.md`：

```markdown
# 后台业务流程

## 登录

客户端                 后端 API              数据库
  │                       │                    │
  │ POST /api/auth/login  │                    │
  │ {email, password}     │                    │
  │──────────────────────→│                    │
  │                       │ 查询用户 by email  │
  │                       │───────────────────→│
  │                       │←─── user record ───│
  │                       │                    │
  │                       │ 验证密码 (bcrypt)  │
  │                       │                    │
  │                       ├─ 成功 ──→ 生成 JWT │
  │                       │          │         │
  │ ←── 200 {token,user} ─│←─────────┘         │
  │                       │                    │
  │                       ├─ 失败              │
  │                       │  ├─ 用户不存在      │
  │                       │  │→ 返回 401       │
  │                       │  ├─ 密码错误        │
  │                       │  │→ 返回 401       │
  │                       │  └─ 错误>上限锁定   │
  │                       │    → 返回 423      │
  │                       │                    │
  │ ←── 401/423 {error} ──│                    │

## 注册

客户端                 后端 API              数据库
  │                       │                    │
  │ POST /api/auth/register │                    │
  │ {nickname,email,pass} │                    │
  │──────────────────────→│                    │
  │                       │ 校验邮箱唯一性     │
  │                       │───────────────────→│
  │                       │←──── count ────────│
  │                       │                    │
  │                       ├─ 已存在 → 返回 409│
  │                       │                    │
  │                       └─ 唯一 →            │
  │                                │           │
  │                                创建用户    │
  │                                │──────────→│
  │                                │←─── user ─│
  │                                生成 JWT    │
  │ ←── 201 {token,user} ──────────┘           │
```

**完整性要求**：必须覆盖所有可能的异常分支。

**门禁**：用户确认流程无误后进入下一步。

---

### 第四步：定义接口与后端功能

**目标**：将后台流程中的关键节点转化为具体的 REST API，形成接口文档初稿。

**流程**：
1. 从 `docs/backend-flows.md` 中提取所有 API 端点
2. 对每个端点描述：URL + Method、请求参数、响应结构、关联后端功能
3. 汇总所有后端需要实现的功能模块

**产出** — `docs/api-list.md`：

```markdown
# 接口清单

## POST /api/auth/login
- 描述：用户登录
- 请求体：
  {
    "email": "user@example.com",
    "password": "secret123"
  }
- 成功响应 200：
  {
    "token": "eyJhbGci...",
    "user": { "id": 1, "name": "Alice" }
  }
- 失败响应：
  401 { "error": "Invalid credentials" }
  423 { "error": "Account locked", "retryAfter": 600 }
- 关联后端功能：AuthService.login()

## POST /api/auth/register
- 描述：用户注册
- 请求体：
  {
    "nickname": "Alice",
    "email": "alice@example.com",
    "password": "secret123"
  }
- 成功响应 201：
  {
    "token": "eyJhbGci...",
    "user": { "id": 2, "name": "Alice", "email": "alice@example.com" }
  }
- 失败响应：
  409 { "error": "Email already registered" }
- 关联后端功能：AuthService.register()

---

# 后端功能模块

| 模块 | 方法 | 说明 |
|------|------|------|
| AuthService | login() | 验证凭据，生成 JWT |
| AuthService | register() | 创建用户，自动登录 |
| AuthService | refreshToken() | 刷新过期 token |
| AuthService | logout() | 注销会话 |
| UserRepository | findByEmail() | 按邮箱查询用户 |
| UserRepository | findById() | 按 ID 查询用户 |
| UserRepository | createUser() | 创建新用户记录 |
| TokenUtil | generateJWT() | 生成带签名的 JWT |
| TokenUtil | verifyJWT() | 验证和解析 JWT |
| RateLimiter | checkLoginAttempts() | 检查登录失败次数 |
```

**门禁**：用户确认接口定义覆盖全部需求后进入下一步。

---

### 第五步：汇总前后端完整功能点

**目标**：生成可供 `pai:spec` 和 `pai:design` 直接使用的功能清单。

**流程**：
1. 从原型图和数据流中提取前端功能点
2. 从后台流程和接口中提取后端功能点
3. 用 ASCII 树形图呈现，标注优先级

**产出** — `docs/feature-points.md`：

```markdown
# 前后端功能点

## 登录注册

### 前端
├─ 用户界面
│  ├─ 登录页 (邮箱/密码/提交/忘记密码/三方登录)
│  ├─ 注册页 (昵称/邮箱/密码/确认密码/提交)
│  ├─ 表单验证 (空值检测、邮箱格式、密码一致性)
│  ├─ 错误提示展示 (空状态→填写提示、错误→标红+文字)
│  └─ 加载状态 (按钮禁用 + 旋转指示器)
│
├─ 交互逻辑
│  ├─ 调用 POST /api/auth/login
│  ├─ 调用 POST /api/auth/register
│  ├─ token 存储到 localStorage
│  ├─ 登录成功后跳转首页
│  ├─ 注册成功后自动登录并跳转
│  └─ 错误时显示具体错误
│
└─ 状态管理
   ├─ 全局用户状态更新
   └─ 页面跳转权限控制

### 后端
├─ API 端点
│  ├─ POST /api/auth/login       (公开)
│  ├─ POST /api/auth/register    (公开)
│  └─ POST /api/auth/refresh     (认证)
│
├─ 业务逻辑
│  ├─ email + password 验证
│  ├─ bcrypt 密码哈希验证
│  ├─ JWT 生成与签名 (HS256, 24h 过期)
│  ├─ 登录失败次数限制 (5 次后锁定 10 分钟)
│  └─ 邮箱唯一性校验
│
├─ 数据模型
│  └─ User { id, nickname, email, password_hash,
│            login_attempts, locked_until, created_at }
│
└─ 安全
   ├─ 密码强度要求 (8+ 字符, 大小写+数字)
   ├─ 登录速率限制 (每分钟 10 次)
   └─ 注册防机器人 (CAPTCHA)

---

## 功能追溯

| 登录页 UI | 用户认证 | Must | US-01 |

| JWT 认证 | 用户认证 | Must | US-03 |

```

## 与 pai:design / pai:spec 的衔接

`pai:story` 完成后，`pai:design` 的阶段 0 自动读取：

- `docs/feature-points.md` → 导入功能点列表，跳过发现式提问
- `docs/api-list.md` → 作为 `pai:spec` 的 API 层 delta spec 输入

后续技能链中的 `pai:spec` 可直接从功能点生成 ADDED requirements：

```markdown
## ADDED Requirements

### Requirement: 用户登录
系统 MUST 提供邮箱+密码登录。
（来源: pai:story feature-points.md → 前端登录页 + 后端 POST /api/auth/login）
```

## 局部迭代

如需修改某一步（如仅更新原型图），通过 `/pai:amend` 管理变更范围。如果功能点有变，可指定步骤编号重新运行（如"重新做第三步"）。

## 注意事项

- **不替代 pai:design** — Story 产出业务需求和原型，Design 产出技术方案
- **不录入代码** — 所有产出都是文档，不生成任何代码
- **可选择性执行** — 用户可指定只做"原型+接口"或只做"功能点"
- **渐进式细化** — 第一次可粗粒度，后续迭代逐步细化
